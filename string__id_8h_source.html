23/**
24 * This represents an identifier (implemented as std::string) of some object.
25 * It can be used for all type of objects, one just needs to specify a type as
26 * template parameter T, which separates the different identifier types.
27 *
28 * The constructor is explicit on purpose, you have to write
29 * \code
30 * auto someid = mtype_id("mon_dog");
31 * if( critter.type->id == mtype_id("mon_cat") ) { ...
32 * \endcode
33 * This allows to find all ids that are initialized from static literals (and not
34 * through the json loading). The can than easily be checked against the json
35 * definition to see whether they are actually defined there (or whether they are
36 * defined only in a mod).
37 *
38 * Example:
39 * \code
40 * struct itype;
41 * using itype_id = string_id<itype>;
42 * struct mtype;
43 * using mtype_id = string_id<mtype>;
44 * \endcode
45 * The types mtype_id and itype_id declared here are separate, the compiler will not
46 * allow assignment / comparison of mtype_id and itype_id.
47 * Note that a forward declaration is sufficient for the template parameter type.
48 *
49 * If an id is used locally in just one header & source file, then feel free to
50 * define it in those files. If it is used more widely (like mtype_id), then
51 * please define it in type_id.h, a central light-weight header that defines all ids
52 * people might want to use. This prevents duplicate definitions in many
53 * files.
54 *
55 * Notes on usage and performance (comparison between ids, ::obj lookup,
56 * assuming default implementation of generic_factory is used):
57 *
58 * `int_id` is fastest, but it can't be reused after game reload. Depending on the loaded
59 * combination of mods `int_id` might point to a different entity and there is no way to tell.
60 * Because of this NEVER define static int ids.
61 * But, it's safe to cache and use int_id within the game session.
62 *
63 * `string_id` is a bit slower than int_id, but it's safe to use the same instance between game reloads.
64 * That means that string_id can be static (and should be for maximal performance).
65 * Comparison of string ids is relatively slow (same as std::string comparison).
66 * for newly created string_id (i.e. inline constant or local variable), first method invocation:
67 * `::id` call is relatively slow (string hash map lookup)
68 * `::obj` lookup is slow (string hash map lookup + array read)
69 * note, after the first invocation, all subsequent calls on the same string_id instance are fast
70 * for old (or static) string_id, second and subsequent method invocations:
71 * conversion to int_id is extremely fast (just returns int field)
72 * `::obj` call is relatively fast (array read by int index), a bit slower than on int_id
73 */

75/**
76 * Type marker that specifies whether string ids of a specific type
77 * are "static" (i.e. "interned", having a single pointer inside) or
78 * "dynamic" (non-interned, having std::string inside).
79 *
80 * By default all string ids are static/interned.
81 * To make a specific string_id type dynamic, add the following specialization:
82 * template<> struct string_id_params<T> {
83 * static constexpr bool dynamic = true;
84 * };
85 * Replace `T` above with concrete type of the string id (see below for examples)
86 */
87template<typename T>
88struct string_id_params {
89 static constexpr bool dynamic = false;
90};
