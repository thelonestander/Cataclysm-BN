602/**
603 * A vehicle as a whole with all its components.
604 *
605 * This object can occupy multiple tiles, the objects actually visible
606 * on the map are of type `vehicle_part`.
607 *
608 * Facts you need to know about implementation:
609 * - Vehicles belong to map. There's `std::vector<vehicle>`
610 * for each submap in grid. When requesting a reference
611 * to vehicle, keep in mind it can be invalidated
612 * by functions such as `map::displace_vehicle()`.
613 * - To check if there's any vehicle at given map tile,
614 * call `map::veh_at()`, and check vehicle type (`veh_null`
615 * means there's no vehicle there).
616 * - Vehicle consists of parts (represented by vector). Parts have some
617 * constant info: see veh_type.h, `vpart_info` structure and
618 * vpart_list array -- that is accessible through `part_info()` method.
619 * The second part is variable info, see `vehicle_part` structure.
620 * - Parts are mounted at some point relative to vehicle position (or starting part)
621 * (`0, 0` in mount coordinates). There can be more than one part at
622 * given mount coordinates, and they are mounted in different slots.
623 * Check tileray.h file to see a picture of coordinate axes.
624 * - Vehicle can be rotated to arbitrary degree. This means that
625 * mount coordinates are rotated to match vehicle's face direction before
626 * their actual positions are known. For optimization purposes
627 * mount coordinates are precalculated for current vehicle face direction
628 * and stored in `precalc[0]`. `precalc[1]` stores mount coordinates for
629 * next move (vehicle can move and turn). Method `map::displace_vehicle()`
630 * assigns `precalc[1]` to `precalc[0]`. At any time (except
631 * `map::vehmove()` innermost cycle) you can get actual part coordinates
632 * relative to vehicle's position by reading `precalc[0]`.
633 * Vehicles rotate around a (possibly changing) pivot point, and
634 * the precalculated coordinates always put the pivot point at (0,0).
635 * - Vehicle keeps track of 3 directions:
636 * Direction | Meaning
637 * --------- | -------
638 * face | where it's facing currently
639 * move | where it's moving, it's different from face if it's skidding
640 * turn_dir | where it will turn at next move, if it won't stop due to collision
641 * - Some methods take `part` or `p` parameter. This is the index of a part in
642 * the parts list.
643 * - Driver doesn't know what vehicle he drives.
644 * There's only player::in_vehicle flag which
645 * indicates that he is inside vehicle. To figure
646 * out what, you need to ask a map if there's a vehicle
647 * at driver/passenger position.
648 * - To keep info consistent, always use
649 * `map::board_vehicle()` and `map::unboard_vehicle()` for
650 * boarding/unboarding player.
651 * - To add new predesigned vehicle, add an entry to data/raw/vehicles.json
652 * similar to the existing ones. Keep in mind, that positive x coordinate points
653 * forwards, negative x is back, positive y is to the right, and
654 * negative y to the left:
655 *
656 * orthogonal dir left (-Y)
657 * ^
658 * -X -------> +X (forward)
659 * v
660 * orthogonal dir right (+Y)
661 *
662 * When adding parts, function checks possibility to install part at given
663 * coordinates. If it shows debug messages that it can't add parts, when you start
664 * the game, you did something wrong.
665 * There are a few rules:
666 * 1. Every mount point (tile) must begin with a part in the 'structure'
667 * location, usually a frame.
668 * 2. No part can stack with itself.
669 * 3. No part can stack with another part in the same location, unless that
670 * part is so small as to have no particular location (such as headlights).
671 * If you can't understand why installation fails, try to assemble your
672 * vehicle in game first.
673 */
