186/**
187 * The *_vector members represent list of alternatives requirements:
188 * alter_tool_comp_vector = { * { { a, b }, { c, d } }
189 * That means: the player needs (requirement a or b) and (requirement c or d).
190 *
191 * Some functions in this struct use template arguments so they can be
192 * applied to all three *_vector members. For example:
193 * check_consistency iterates over all entries in the supplied vector and calls
194 * check_consistency on each entry.
195 * If called as <code>check_consistency(tools)</code> this will actually call
196 * tool_comp::check_consistency. If called as
197 * <code>check_consistency(qualities)</code> it will call
198 * quality_requirement:check_consistency.
199 *
200 * Requirements (item_comp, tool_comp, quality_requirement) must have those
201 * functions:
202 * Load from an entry of a json array:
203 * void load(const JsonValue &value);
204 * Check whether the player has fulfills the requirement with this crafting
205 * inventory (or by mutation):
206 * bool has(const inventory &crafting_inv) const;
207 * A textual representation of the requirement:
208 * std::string to_string() const;
209 * Consistency check:
210 * void check_consistency(const std::string &display_name) const;
211 * Color to be used for displaying the requirement. has_one is true of the
212 * player fulfills an alternative requirement:
213 * nc_color get_color(bool has_one, const inventory &crafting_inv) const;

412// Sometimes the requirement_data is problematic, because it has overlapping
413// requirements. For example, a survivor telescope requires
414// 1 high-quality lens
415// 1 high-quality lens OR 1 small high-quality lens
416// If there is just one high-quality lens in the available inventory then it's
417// hard to correctly detect that these requirements are impossible to satisfy.
418// In general, determining craftability is equivalent to boolean
419// satisfiability, and thus NP-hard.
420//
421// In practice, we don't expect recipes to have too much overlap, so this issue
422// should be tractable.
423//
424// However, to avoid keeping additional state during the process of searching
425// for components, we don't make the component search more complex, instead we
426// make the requirements more complex. We replace each requirement_data with a
427// deduped_requirement_data, which contains a selection of alternative
428// requirement_data objects, each of which contains no overlapping
429// requirements.
430//
431// For the majority of recipes, there are no overlaps, and this will be
432// essentially equivalent to just a requirement_data. However, for the few
433// problematic recipes this allows us to calculate crafting requirements more
434// accurately.
435class deduped_requirement_data
436{
437 public:
438 using alter_item_comp_vector = requirement_data::alter_item_comp_vector;
